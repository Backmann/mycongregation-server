## Conventions and gotchas learned today

These are real, hard-won insights that should not be re-discovered next session.

### 1. Husky in Docker production builds

`"prepare": "husky"` breaks `npm ci --omit=dev` because npm runs `prepare` after install, but husky is in devDependencies and isn't there.

**Fix:** use `"prepare": "husky || true"`. In dev, husky runs normally. In prod with no husky binary, `|| true` swallows the 127 exit. Standard husky-in-Docker workaround.

### 2. ESLint --fix runs silently and accumulates drift

`npm run lint` in server's package.json has `--fix`. Each run silently modifies files (removing redundant casts, fixing imports, etc.). These mods sit unstaged unless explicitly committed.

**Pattern observed today:** Multiple `npm run lint` runs during P1 #17 categorization quietly modified 4 files with `no-unnecessary-type-assertion` auto-fixes. Captured separately in `3c5bc0f`.

**Implication:** running `npm run lint` without `--fix` is safer for diagnostic-only runs. Or always commit drift immediately. Or remove `--fix` from the package.json script and run `npm run lint:fix` explicitly when wanted (cleaner separation).

### 3. Prettier baseline drift was a silent mine before husky

Before today, the repo had `.prettierrc` but never ran prettier on the source tree. Files drifted from config. Today's accidental `npm run format` run (probably via an editor or earlier tool) reformatted 45 files. Without husky to enforce on commit, this kind of mass drift recurs every time someone runs format.

**Fix in place:** P1 #19 husky + lint-staged on every commit. Plus P1 #18 commit (`258554a`) establishes the new baseline.

**Future:** if devs adopt different prettier versions or configs, drift comes back. Pin prettier version in package.json (already done at `^3.4.2` — close enough).

### 4. CLDR plural awareness in i18n drift checks

Russian has 4 plural categories (`_one`, `_few`, `_many`, `_other`); English/German only 2 (`_one`, `_other`); Arabic adds `_zero`/`_two`. A naive "all keys must exist in all locales" check produces false-positive drift on `_few`/`_many` keys.

**Fix:** `scripts/check-i18n-drift.mjs` exempts `_zero`, `_two`, `_few`, `_many` from drift detection. `_one` and `_other` are still required everywhere.

### 5. MINGW64 bracketed paste mode hazard

When pasting a multi-line block into MINGW64 bash, the terminal sometimes wraps content with `\e[200~` and `\e[201~`. If bash doesn't handle bracketed paste correctly, you get errors like `bash: [200~cd: command not found`.

**Fix:** `bind 'set enable-bracketed-paste off'`. Type by hand, paste-then-Enter cleanly. Or add to `~/.inputrc` permanently:
```
set enable-bracketed-paste off
```

### 6. SSH session interaction trap

If you paste a multi-line block where the first line is `ssh myserver` and the rest are local commands, the ssh process accepts the rest as stdin and tries to run them on the remote. Worked twice today during P0 #10.

**Pattern:** always wait for the remote prompt (`root@ubuntu-...#`) before sending more commands. Or for scripted ops, use `ssh myserver bash -c 'commands'`.

### 8. ESLint config override pattern for spec files

For NestJS + jest projects, the cleanest way to handle the inevitable `any` heavy mocks in tests is a flat-config override:

```js
{
  files: ['**/*.spec.ts', '**/*.test.ts'],
  rules: {
    '@typescript-eslint/no-unsafe-member-access': 'off',
    '@typescript-eslint/no-unsafe-assignment': 'off',
    '@typescript-eslint/no-unsafe-return': 'off',
    '@typescript-eslint/no-unsafe-call': 'off',
    '@typescript-eslint/no-unsafe-argument': 'off',
    '@typescript-eslint/unbound-method': 'off',
    '@typescript-eslint/require-await': 'off',
  },
},
```

This is the typescript-eslint docs' recommended pattern for jest projects. Applied today in `8999d2a` — wiped 302 of 395 errors.

### 9. Production reboot procedure (proven today)

For Hetzner CX22 with Docker Compose containers (mycongregation + 30sec):

```bash
# 1. Pre-flight
ssh myserver
docker ps --format 'table {{.Names}}\t{{.Status}}'

# 2. Backups (using actual POSTGRES_USER, not assumed `postgres`)
mkdir -p /root/pre-reboot-backups && cd /root/pre-reboot-backups
TS=$(date +%Y%m%d_%H%M%S)
docker exec mycongregation-postgres pg_dumpall -U mycongregation_user > mycongregation_${TS}.sql
docker exec 30sec-postgres pg_dumpall -U sec30user > 30sec_${TS}.sql

# 3. Apt update + upgrade (non-interactive, safe for /etc conflicts)
apt update
DEBIAN_FRONTEND=noninteractive apt-get upgrade -y \
  -o Dpkg::Options::="--force-confdef" -o Dpkg::Options::="--force-confold"

# 4. Held-back packages (esp. kernel meta)
DEBIAN_FRONTEND=noninteractive apt-get dist-upgrade -y \
  -o Dpkg::Options::="--force-confdef" -o Dpkg::Options::="--force-confold"

# 5. Reboot
sync && reboot
# SSH drops — wait ~60-90s

# 6. Verify (from local)
ssh myserver  # may need 1-2 retries
uname -r      # confirm new kernel
docker ps     # all containers up
exit
curl -sI https://api.mycongregation.org/api/health
curl -sI https://mycongregation.org
curl -sI https://30sec.org
```

Today this took ~60-90 sec total downtime. Docker auto-restart worked flawlessly (`restart: unless-stopped` policies in compose files).

### 10. Container postgres users are NOT "postgres"

Today's gotcha: `pg_dumpall -U postgres` fails on both mycongregation-postgres and 30sec-postgres because the actual `POSTGRES_USER` env var was set to `mycongregation_user` and `sec30user` respectively, and the default postgres role doesn't exist.

**Discovery command:** `docker exec <container> printenv POSTGRES_USER`

## Workflow conventions reaffirmed today

- **Tarball delivery** via Claude → SCP via Downloads → extract → commit. Established pattern, worked well today.

- **Selective `git add`**, not `git add -A`. App repo `.env` was tracked all day; with .env now untracked and gitignored, this is less of a problem going forward.
- **Pre-commit prettier on server only** (app repo doesn't have prettier setup yet; Expo Lint runs in CI). If app drift becomes a problem, add husky to app repo too.
