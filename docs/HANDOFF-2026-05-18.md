`rejectUnauthorized: false` because the cert is self-signed; password still authenticates. Server-side prerequisites applied via `phase-4b-server-setup.sh` (idempotent bash script):

## Conventions and gotchas learned in this session

### Git workflow (lessons from today)

- **Commit on the branch BEFORE `git checkout main`** — otherwise `git checkout` carries uncommitted changes to main, leaving them unstaged. `git merge phase-branch` reports "Already up to date" because no commits were made.
- **Don't paste multi-line commit messages directly into bash** — bash tries to execute `feat(users): ...` as a command and fails with `syntax error near unexpected token '('`. Use `cat > /tmp/commit-msg.txt << 'COMMITMSG' ... COMMITMSG` heredoc, then `git commit -F /tmp/commit-msg.txt`.
- **Selective `git add` when working tree has unrelated changes** — `.env` and `.gitignore` had Lionel-side edits that shouldn't be in feature commits. Always `git status` after `git add` and before `git commit` to verify what's staged.
- **WATCHOUT: `rm -rf scripts` is too broad** when the scripts/ directory has BOTH the one-shot helper script AND tracked dev tools. Today this accidentally deleted 4 tracked files (inspect-mwb.ts, inspect-wt.ts, parse-mwb.ts, parse-wt.ts) — they were recovered with `git restore scripts/`, but next time use `rm scripts/<specific-script>.mjs` instead.
- **A branch can already exist from a previous attempt** — `git checkout -b foo` failed with "already exists" but `git status` showed Lionel already on that branch. Git proceeds anyway. Just be aware.

### Curl smoke testing

- **Substitute placeholders in templates** — when Claude provides `password:"<твой текущий>"`, replace with the real value. Otherwise login fails (401), `AT` becomes literal `"null"`, downstream curl returns 401 from `JwtAuthGuard` rejecting `bearer null`. Looks like endpoint failure but isn't.
- **401 vs 404 as deployment signal** — in NestJS, route matching happens before guards. 401 = "route exists, auth rejected"; 404 = "route doesn't exist".
- **Class-validator runs before auth/service** — testing endpoints with invalid DTOs returns 400 from `ValidationPipe` before any DB query. To verify DB connection through endpoint, make sure DTO is valid: e.g. for login, password must be ≥8 chars (matches `@MinLength(8)`) — only then does `bcrypt.compare` run after `findOneByEmail`.
- **`pg_stat_ssl` includes the diagnostic psql session** — filter with `pid != pg_backend_pid()` for true client-backend counts.
- **TypeORM pool can be idle** — `pg_stat_ssl` snapshot may show 0 client connections even though the app is healthy. Trigger activity with a real DB-hitting request first (login with valid DTO is a clean trigger).

### Build / deploy gotchas (carried forward + new)

- **Metro/Babel cache + env vars**: when env vars change, `rm -rf .expo node_modules/.cache dist` is mandatory or stale bundles ship. App-side CI workflow does this.
- **`.env.production` vs `.env`**: Expo loads both when `NODE_ENV=production`; `.env.production` overrides `.env`. Inline `VAR=value command` may not work on MINGW64 bash — use `export` or `.env.production`.
- **`EXPO_PUBLIC_*` env vars are NOT secrets** — inlined into JS bundle at build time, exposed to every client browser. Designed for client-side configuration (API URLs, VAPID public keys, analytics IDs). Real secrets live on the server.
- **Bind mount inode**: never `mv` the mount-point directory. Docker bind mounts attach to inode at container start. Replace contents in-place.
- **Heredoc inside `ssh "..."` on MINGW64** tends to mangle. Pipe via `cat <<EOF | ssh ...` or one-liners with `psql -c "..."`.
- **`bash !` history expansion**: bash treats `!` in double-quoted strings as history expansion. Passwords/values with `!` need single quotes or `set +H`.
- **MINGW64 terminal capacity**: very long single-command-block inputs may truncate. Split into 3-5 commands per batch.
- **Symlink/junction quirk on Windows**: `~/congmap/app/locales/` resolves to `~/congmap/app/app/locales/` via directory junction. Writing to one updates the other.

### Code conventions

- **TypeORM `synchronize: false`** — all schema changes are manual SQL migrations under `src/migrations/`.
- **Multi-tenancy** — every entity has `congregationId`. Queries filter by `@TenantId()` decorator. For cross-tenant access attempts, return **404 not 403** to avoid leaking existence.
- **Soft-delete** — `deletedAt` column, excluded by default in queries.
- **Push channels** — Expo native AND Web Push sent from same `sendStatusChange` call; `excludeUserId` is the publisher's own `userId`.

### Audit log patterns

- `auditLog.logUpdate(...)` — auto-diff between `before`/`after`; no-op if all `fields` unchanged. Stores actual values.
- `auditLog.logCreate(...)` — for entity creation events; stores `after` verbatim, `before = null`. Callers responsible for omitting sensitive fields.
- `auditLog.logRawUpdate(...)` — caller specifies `changedFields` explicitly; stores both `before` and `after` verbatim. **Used when masking sensitive values** like `{ passwordHash: '<redacted>' }` (auto-diff would treat equal masked values as "unchanged").
- **`actorUserId === entityId`** marks the change as a **self-action** (user changing own data). Distinguishes admin reset from self-service password change in forensic analysis.

### Authorization patterns

- `@UseGuards(RolesGuard)` and `@Roles(UserRole.X)` at the **class level** when entire controller is admin-only (e.g. `UsersController`). At the **method level** for mixed-access controllers.
- `@TenantId()` param decorator extracts `req.user.congregationId` from JWT payload.
- `JwtAuthGuard` is global via `APP_GUARD` in `AuthModule`. `@Public()` opts out (used only for `bootstrap`, `login`, `refresh`).
- **Bcrypt error responses:** 400 BadRequest for "current password wrong" (avoids triggering client refresh interceptor). 401 reserved for token-related auth failures.

### Encryption at rest (Phase L Phase 1)

- **CryptoService** (singleton via `crypto-store.ts`) is initialized at NestJS bootstrap from `KEK_BASE64` env var (32 bytes, base64). Bootstrap fails fast if KEK is missing/malformed — prevents app starting in a state where it could overwrite encrypted data with a different key.
- **Format on disk**: `enc:v1:<base64(iv)>:<base64(ciphertext||authTag)>`. AES-256-GCM, 96-bit random IV per write, 128-bit auth tag. `enc:v1:` prefix allows mixed-mode reads during migrations (`CryptoService.decrypt` passes through non-prefixed strings unchanged).
- **`encryptedTransformer`** is a TypeORM `ValueTransformer` — `to(value)` encrypts on write, `from(value)` decrypts on read. Applied via `@Column({ type: 'text', nullable: true, transformer: encryptedTransformer })`.
- **Migration `WidenEncryptedColumns1779000000000`** converted size-bounded columns (`mobile_phone varchar(32)`, `email varchar(255)`) to unbounded `text` because ciphertext is ~85 bytes longer than plaintext.
- **AuditLog encryption is critical** — without `encryptedTransformer` on `beforeJson`/`afterJson`, the audit log would mirror plaintext copies of encrypted values via auto-diff. Today's regression test asserts both columns retain the transformer.

### TLS / Postgres (Phase L Phase 4B)

- **Self-signed cert** at `/root/mycongregation/postgres-tls/server.{crt,key}`, CN=postgres, 10y validity.
- **Owner must be 70:70** for Alpine Postgres image (Debian uses 999).
- **TypeORM `ssl: { rejectUnauthorized: false }`** for self-signed certs. Password still authenticates the connection.
- **Verification query**: `SELECT pid, ssl FROM pg_stat_ssl JOIN pg_stat_activity USING (pid) WHERE backend_type = 'client backend' AND pid != pg_backend_pid()`. Filter out psql's own pid; require actual app traffic (TypeORM pool can be idle).
- **`env_file:` in compose loads at container CREATE time**, baked into container. Subsequent `.env` edits don't reach the container until next recreate. CI deploy recreates, so changes propagate on next push.

### Client UX patterns established for admin work

- **Admin-only screens live under `/profile/`** (not a separate tab). `public-talks.tsx` set the pattern; Phase 2 (Responsibility management) etc. sit alongside.
- **`usePermissions()` hook** in `lib/permissions.ts` — single source of UI-side capability flags. Server remains authoritative.
- **Native `Alert.alert` for destructive confirmations** — no extra deps. `{ text, style: 'destructive', onPress }` + `{ text: cancel, style: 'cancel' }`.
- **`Modal` + `KeyboardAvoidingView` + `SafeAreaView` + `presentationStyle="pageSheet"`** for form modals. Header with Cancel / Title / Submit, submit disabled when validation fails or mutation pending.
- **react-query mutations** with `onSuccess: () => qc.invalidateQueries(...)` to refresh lists, `onError: (err) => Alert.alert(t('common.error'), extractErrorMessage(err))` to surface server errors verbatim.
