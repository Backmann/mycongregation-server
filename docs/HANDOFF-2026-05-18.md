## Conventions and gotchas learned in this session

### Git workflow (bit us twice)

- **Commit on the branch BEFORE `git checkout main`**. Otherwise `git checkout` carries uncommitted changes onto main, where they sit unstaged, and `git merge phase-branch` reports "Already up to date" (because no commits were made on the branch). Result: code is in working tree on main, but never committed or pushed, and CI never sees it. The 404s from the post-deploy smoke test were the first sign.
- **Don't paste multi-line commit messages directly into bash.** Bash will try to execute `feat(users): ...` as a command and fail with `syntax error near unexpected token '('`. Use `cat > /tmp/commit-msg.txt << 'COMMITMSG' ... COMMITMSG` heredoc, then `git commit -F /tmp/commit-msg.txt`.
- **Selective `git add` when working tree has unrelated changes.** Today's app-repo commit needed `git add lib/ app/ locales/` explicitly because `.env` and `.gitignore` had Lionel-side edits that shouldn't be in the same commit. Always run `git status` after `git add` and before `git commit` to verify what's staged.
- **A branch can already exist from a previous attempt.** `git checkout -b admin-users-ui` failed with "already exists" but the next `git status` showed Lionel was already on that branch — git proceeds with the existing branch. Generally fine; just be aware.

### Curl smoke testing

- **Substitute placeholders in templates.** When Claude provides `password:"<твой текущий>"`, that's a placeholder — replace with the real value. Otherwise login fails (401), `AT` becomes the literal string `"null"`, and downstream curl calls return 401 from `JwtAuthGuard` rejecting `bearer null`. This looks like an endpoint failure but isn't.
- **401 vs 404 as deployment signal.** In NestJS, route matching happens before guards. So 401 means "route exists, auth rejected"; 404 means "route doesn't exist". When verifying a fresh deploy, distinguish these.

### Build / deploy gotchas (carried forward)

- **Metro/Babel cache + env vars**: when env vars change, `rm -rf .expo node_modules/.cache dist` is mandatory or stale bundles ship. App-side CI workflow does this automatically.
- **`.env.production` vs `.env`**: Expo loads both when `NODE_ENV=production` (which `expo export -p web` sets by default). `.env.production` overrides `.env`. **Inline `VAR=value command` may not work on MINGW64 bash** — use `export` or `.env.production`.
- **`EXPO_PUBLIC_*` env vars are NOT secrets.** They are inlined into the JS bundle at build time and exposed to every client browser. Designed for client-side configuration (API URLs, VAPID public keys, analytics IDs). Real secrets live on the server.
- **Bind mount inode**: never `mv` the mount-point directory (`/var/www/mycongregation/`). Docker bind mounts attach to inode at container start, not path. Replace contents in-place: `rm -rf ./* && tar xzf`. App CI does this correctly.
- **Heredoc inside `ssh "..."` on MINGW64**: tends to mangle. Pipe SQL via `cat <<EOF | ssh ...` or use one-liners with `psql -c "..."`. (Heredoc locally on MINGW64 bash is fine — only the ssh-quoted variant has issues.)
- **`bash !` history expansion**: bash treats `!` in double-quoted strings as history expansion. Passwords/values with `!` need single quotes or `set +H` disable.
- **MINGW64 terminal capacity**: very long single-command-block inputs may not fit. Split into 3-5 commands per batch when investigating.
- **Symlink/junction quirk on Windows**: `~/congmap/app/locales/` resolved to `~/congmap/app/app/locales/` for git tracking (one is a directory junction to the other). Writing to one updates the other. Confused us briefly in the diff output.

### Code conventions

- **TypeORM `synchronize: false`** — all schema changes are manual SQL migrations under `src/migrations/`.
- **Multi-tenancy** — every entity has `congregationId`. Queries filter by `@TenantId()` decorator. For cross-tenant access attempts, return **404 not 403** to avoid leaking existence.
- **Soft-delete** — `deletedAt` column, excluded by default in queries.
- **Push channels** — Expo native AND Web Push are sent from the same `sendStatusChange` call; `excludeUserId` is the publisher's own `userId` (so they don't get notified about their own status change).

### Audit log patterns established this session

- `auditLog.logUpdate(...)` — auto-diff between `before`/`after`; no-op if all `fields` are unchanged. Stores actual values.
- `auditLog.logCreate(...)` — for entity creation events; stores `after` verbatim, `before = null`. Callers responsible for omitting sensitive fields.
- `auditLog.logRawUpdate(...)` — caller specifies `changedFields` explicitly; both `before` and `after` stored verbatim. **Used when masking sensitive values** like `{ passwordHash: '<redacted>' }` (auto-diff would treat equal masked values as "unchanged").
- **`actorUserId === entityId`** in an audit row marks the change as a **self-action** (user changing their own data). Distinguishes admin reset from self-service password change in forensic analysis.

### Authorization patterns established this session

- `@UseGuards(RolesGuard)` and `@Roles(UserRole.X, ...)` at the **class level** when the entire controller is admin-only (e.g. `UsersController`). At the **method level** for mixed-access controllers.
- `@TenantId()` param decorator extracts `req.user.congregationId` from the JWT payload — every admin operation should be scoped through it.
- `JwtAuthGuard` is global via `APP_GUARD` in `AuthModule`. All routes require auth by default; `@Public()` opts out (used only for `bootstrap`, `login`, `refresh`).
- **Bcrypt error responses:** 400 BadRequest for "current password wrong" (not 401, to avoid triggering client refresh interceptor). 401 reserved for token-related auth failures.

### Client UX patterns established for admin work

- **Admin-only screens live under `/profile/`** (not a separate tab). The existing `public-talks.tsx` set the pattern; Phase 2 (Responsibility management) etc. will sit alongside.
- **`usePermissions()` hook** in `lib/permissions.ts` — single source of UI-side capability flags. Server remains authoritative. Pattern is to add new flags here as new capabilities ship, then check them at render time.
- **Native `Alert.alert` for destructive confirmations** — no extra deps needed. Pattern: `{ text, style: 'destructive', onPress }` for the destructive action, plus `{ text: cancel, style: 'cancel' }`.
- **`Modal` + `KeyboardAvoidingView` + `SafeAreaView` + `presentationStyle="pageSheet"`** for form modals. Header bar with Cancel (left) / Title (center) / Submit (right), submit button disabled when validation fails or mutation pending.
- **react-query mutations** with `onSuccess: () => qc.invalidateQueries(...)` to refresh lists after writes, and `onError: (err) => Alert.alert(t('common.error'), extractErrorMessage(err))` to surface server errors verbatim (including server-side authorization errors like "Cannot demote the last active admin").

**Option A — Phase L Data Protection (1-2 days dedicated focus)**

Encryption at rest. `KEK_BASE64` already on server.
